# 1.2.3 MCP 与 1.2.4 Skill 章节扩写设计说明

## 背景

当前 `docs/claude_code_arch.md` 中的 `1.2.3 MCP：工具体系的无限扩展口` 与 `1.2.4 Skill 系统：把工作流固化成命令` 都已经具备基础信息，但问题也很接近：

- 能让读者知道“它们是什么”
- 还不足以说明“它们为什么在 Claude Code 的整体架构里重要”
- 与前一节 `1.2.2 工具体系` 的衔接还不够紧
- 与后续上下文管理、多 Agent、Hooks 等机制的关系也没有完全展开

用户希望这两节都做增强，并且保持短小对称：两节都扩为中等篇幅，分析风格一致，不出现一节很厚、一节很薄的失衡。

本次扩写的核心目标，是把 `1.2.3` 和 `1.2.4` 从“能力介绍”升级为“扩展接口分析”：

- **MCP** 讲清楚它如何把外部服务接成 Claude Code 运行时可统一调度的能力单元
- **Skill** 讲清楚它如何把团队内部流程固化成可复用、可调用的工作流资产

## 写作目标

新版本的两节需要完成三件事：

### 1. 与 `1.2.2` 构成自然递进

`1.2.2` 已经说明了 Claude Code 的工具不是简单列表，而是带权限、调度属性和结果裁剪策略的执行节点。`1.2.3` 与 `1.2.4` 应继续回答：

- 外部能力怎么进入这套执行节点系统？ → `1.2.3 MCP`
- 内部工作流怎么进入这套执行节点系统？ → `1.2.4 Skill`

换句话说，这两节应该成为 `1.2.2` 的自然后续，而不是各自孤立的功能说明。

### 2. 保持“分析报告型”而非“使用手册型”

文本应继续沿用全文当前风格：

- 先给判断，再解释机制
- 讲清设计取舍和架构位置
- 用少量配置片段或结构块帮助理解
- 不写成 API 文档，也不写成逐项功能罗列

### 3. 两节保持对称

两节都建议扩到中等篇幅，长度大致相近，各自都应包含：

1. 开场判断
2. 在整体架构中的角色
3. 运行机制或接入机制
4. 与另一节的边界
5. 收束结论

目标是让读者读完后，对这两节形成清晰分工：

- MCP 是“接系统”
- Skill 是“固化流程”

## 推荐总体策略：接口定位型

本次推荐把 `1.2.3` 和 `1.2.4` 都写成 Claude Code 的两种扩展接口，但接口方向不同：

- `1.2.3 MCP`：面向外部系统的扩展接口
- `1.2.4 Skill`：面向内部工作流的扩展接口

这会让章节结构非常自然：

- `1.2.2`：工具体系本身如何被设计
- `1.2.3`：外部服务如何被接入工具体系
- `1.2.4`：内部流程如何被固化进执行体系

## `1.2.3 MCP` 的设计

### 开场定位

开头不应只说“MCP 是无限扩展口”，而要把这句话落到架构层面。正文应覆盖这些判断点：

- MCP 的意义不只是“可以接第三方工具”
- 它把外部系统转换成 Agent Loop 能统一调度的能力单元
- 内置工具解决的是通用操作，MCP 解决的是企业环境里持续出现的专有系统接入问题

这段的作用，是把 MCP 从“可选插件能力”抬升为“工具体系的接口层”。

### 第一部分：MCP 在整体架构里的角色

明确三点：

1. MCP 不是单个工具，而是一层协议适配与运行时注册机制
2. 它负责把外部 server 暴露的 tools/resources 纳入 Claude Code 的统一执行面
3. 它让 Claude Code 的能力上限不再由发布包中的内置工具决定，而由运行环境决定

这一部分最终需要落出一个清晰判断：65 个内置工具是基础设施，而 MCP 是 Claude Code 进入企业环境的关键接口层。

### 第二部分：接入机制与多传输协议

当前文档已经列出了 5 种协议，但需要补出“为什么要有这么多种”。

建议分别解释：

- `Stdio`：适合本地脚本与本地进程，是最常见的开发机接入方式
- `SSE`：适合轻量远程服务，部署成本低
- `WebSocket`：适合持续双向连接场景
- `HTTP`：适合标准化流式 HTTP 服务
- `SDK Control`：适合进程内嵌入或更深度集成

这一段的重点，不是枚举协议，而是说明 Claude Code 的 MCP 客户端不是 demo 级实现，而是认真把“本地工具、远程服务、嵌入式集成”都纳入了统一协议面。

### 第三部分：认证与安全边界

这一部分应强调：MCP 不是裸连。

应明确提到：

- OAuth 2.0
- PKCE
- token 自动刷新
- `X-MCP-Scope`

需要表达的核心判断是：Claude Code 的 MCP 设计不是只考虑“能连上”，而是已经考虑了企业内部服务接入的安全现实。这意味着它不是玩具扩展口，而是具备进入受控组织环境的协议基础。

### 第四部分：动态工具加载

这是 `1.2.3` 的关键段落，应写清：

1. MCP server 暴露工具定义
2. Claude Code 在运行时发现并加载这些定义
3. 对模型来说，这些工具和内置工具几乎没有调用方式上的差别

这里最终应得出的结论是：一旦接入 MCP，Claude Code 的“工具列表”就不再是编译期常量，而变成运行时可扩展集合。

### 第五部分：与 Skill 的边界

为后面的 `1.2.4` 做承接，建议在结尾提前埋一个边界判断：

- MCP 接的是“外部能力”
- Skill 固化的是“内部方法”

这样读者进入下一节时，会自然知道两者不是竞争关系，而是不同方向的扩展层。

### 收束结论

收束段应明确：MCP 让 Claude Code 不再受限于发行包里写死的能力，而是可以把企业现场里分散的服务、搜索、工单、知识库和内部系统统一拉进 Agent Loop 的调度面。

## `1.2.4 Skill` 的设计

### 开场定位

当前“把工作流固化成命令”的标题方向是对的，但正文可以更进一步。开场应覆盖这些判断点：

- Skill 解决的不是“少打一段提示词”这么简单
- 它要把团队里反复出现的流程、规则和判断标准固化成可调用资产
- 与 MCP 相比，Skill 处理的是内部方法，而不是外部系统接入

这段的作用，是把 Skill 从“Prompt 模板”抬高到“工作流资产”。

### 第一部分：Skill 在架构里的角色

这一部分要讲清楚 Skill 不是协议，而是工作流封装层：

1. 它面向的是团队内部知识与流程，而不是外部服务
2. 它的目标是降低高频操作的重复描述成本
3. 它让“代码审查”“提交前检查”“写文档”“跑测试”这类流程从临时对话变成稳定入口

这一部分最后需要落出一个明确判断：Skill 把一次次临时提示词，变成了可复用、可传播、可维护的团队资产。

### 第二部分：Skill 的执行机制

需要保留现有 `settings.json` 的例子，但不能只停留在“定义了什么”。应把链路补出来：

1. 用户在 `settings.json` 中声明技能
2. `SkillTool` 查找对应 skill
3. 如果是文件型 skill，读取内容并注入上下文
4. 如果是命令型 skill，执行对应命令
5. 最终效果是把“提示 + 约束 + 步骤”一起注入当前任务

这一段要强调：Skill 的关键不在于格式，而在于它改变了 AI 每次工作的起点。

### 第三部分：Skill 的可维护性与组织沉淀价值

为了和 MCP 的“协议 / 认证 / 动态加载”细节形成对称，Skill 也需要一段更具体的机制分析，不能只停留在“像 prompt template 一样可复用”。这一段建议明确以下几点：

1. Skill 可以被放进项目或团队配置中持续维护，而不是只存在于某次对话里
2. 它可以承载团队约定的检查步骤、交付格式、审查口径和命名规范
3. 它让新成员或 AI 在进入任务时直接获得同一套起手方式，减少流程漂移
4. 它把“谁记得这套做法”转成“系统里是否有这个 Skill”

这段的作用，是把 Skill 写成真正的组织级能力，而不只是“好用的命令别名”。

### 第四部分：为什么 Skill 比普通 Prompt Template 更强

这一段是当前版本最缺的内容，建议重点补上。

要明确区分：

- Prompt template：静态文本片段
- Skill：带入口、带约束、可长期沉淀的工作流单元

Skill 的强点在于：

- 可以绑定团队术语和规则
- 可以与工具、任务系统、权限和日常流程搭配
- 不依赖某个成员每次重新讲同一套做法

这一段最后应得出的结论是：Skill 不是给模型“更多提示”，而是给团队“更稳定的做事方式”。

### 第五部分：与 MCP 的边界

这一段要和 `1.2.3` 对称，清晰给出边界：

- MCP 是接系统
- Skill 是固化流程
- 一个解决“接什么能力”
- 一个解决“按什么方法做”

这样可以防止读者把两者都理解成“扩展能力”的同义词。

### 收束结论

收束段应明确：Skill 让 Claude Code 不只是会做事，还能按团队熟悉的方式做事；它固化的不是工具能力，而是组织内部反复验证过的工作流。

## 篇幅与对称性要求

两节建议控制在接近的体量：

- `1.2.3 MCP`：约 45~70 行
- `1.2.4 Skill`：约 45~70 行

这样既能明显强于当前版本，又不会压过后面的 `1.2.5 上下文管理`。

## 与相邻章节的边界

为了避免重复，需控制边界如下：

- **与 `1.2.2 工具体系` 的边界**：这里只讲“如何扩展工具体系”，不重复工具分类本身
- **与 `1.2.5 上下文管理` 的边界**：Skill 可以提上下文注入，但不展开上下文压缩与五层机制
- **与 `1.2.6 任务系统` 的边界**：Skill 可提流程固化，但不展开任务持久化机制
- **与 `1.3.4 多 Agent 协调` 的边界**：MCP / Skill 可提与多 Agent 协同的潜力，但不展开 Agent 基础设施细节

最终替换到文档里的 `1.2.3` 和 `1.2.4` 应满足：
