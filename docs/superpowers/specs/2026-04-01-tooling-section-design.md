# 1.2.2 工具体系章节扩写设计说明

## 背景

当前 `docs/claude_code_arch.md` 的 `1.2.2 工具体系：65 个工具的分类逻辑` 章节信息量偏低，主要停留在工具清单和粗粒度二分法（执行类 / 协调类）层面。它能让读者知道 Claude Code 大概有哪些工具，但还不足以解释：

- 为什么这些工具要这样分类
- 工具分类与运行时调度、权限系统、上下文预算之间的关系
- 为什么 Claude Code 的工具体系不是普通的“函数调用列表”

用户希望将本节增强为“分析报告型”内容，保持全文现有的高密度、强结论、源码解构式写法，同时补足以下 4 个主题：

1. 工具分类与职责划分
2. 权限检查链的完整流程
3. 大结果处理与临时文件机制
4. 工具并发安全控制策略

目标篇幅为中等扩写：从当前约 20 行扩充到约 80~120 行，讲透机制，但避免与 `1.2.8 流式执行引擎` 和 `1.3.1 权限系统` 出现大段重复。

## 写作目标

新版本的 `1.2.2` 需要完成三件事：

### 1. 从“工具目录”升级为“工具架构”

不再只回答“Claude Code 有哪些工具”，而是回答“Claude Code 为什么把工具做成现在这样”。

核心观点应当是：

> Claude Code 的工具体系不是 65 个平铺能力，而是一组带有权限、并发、结果规模、执行副作用等元信息的受控执行单元。

### 2. 从工具视角串联其他核心模块

本节要成为 `1.2.1 Agent Loop`、`1.2.8 流式执行引擎`、`1.3.1 权限系统` 之间的桥梁章节。它不需要重复那些章节的实现细节，但要明确说明：

- Agent Loop 为什么能安全地调度工具
- 流式执行为什么需要区分并发安全与否
- 权限系统为什么必须跟工具分类强耦合
- 大结果为什么不能直接原样回灌上下文

### 3. 保持分析报告口吻

文本风格应延续全文特点：

- 先给结论，再给结构和伪代码
- 多使用“这意味着什么”式判断句
- 既讲机制，也讲设计取舍
- 不写成 API 手册，也不写成逐函数源码注释

## 推荐结构

### 开场定位

用一段短引子重写当前章节开头，直接抬高本节的分析层级。建议表达为：

> Claude Code 的 65 个工具不能只按“能做什么”分类，更关键的是“执行时会触发什么控制逻辑”。一个工具是否有副作用、是否可能产生超大输出、是否允许并发、是否承担工作流协调职责，决定了它在系统里的真实位置。

这段话的作用是提前告诉读者：本节要讲的不是工具列表，而是工具抽象背后的运行时规则。

### 第一部分：分类逻辑不是功能目录，而是运行时属性划分

这一部分先解释“分类标准”本身。

建议引入 4 个分析轴：

1. **能力域**：文件、搜索、Shell、Web、Notebook、任务、Agent、MCP
2. **副作用等级**：只读 / 本地可写 / 外部执行
3. **调度属性**：并发安全 / 必须串行
4. **控制角色**：执行类 / 协调类 / 扩展桥接类

这里的重点不是列举，而是强调：Claude Code 工具的真实分类标准并不是 UI 层看到的名称，而是运行时如何被调度、审批、裁剪和组合。

建议用一句收束判断：

> 表面上看是 65 个工具，底层其实是一套带元数据的执行节点系统。

### 第二部分：职责分层

这一部分在现有“执行类 / 协调类”基础上，扩展为三层：

#### 1. 执行类工具

直接对文件系统、终端、网络或笔记本内容进行读写与查询，产出可观察结果：

- 文件与搜索：`FileRead` / `FileEdit` / `FileWrite` / `GlobTool` / `GrepTool`
- Shell：`BashTool` / `PowerShellTool`
- Web：`WebFetchTool` / `WebSearchTool`
- 数据科学：`NotebookEdit`

#### 2. 协调类工具

不直接产出业务结果，而是管理 AI 的工作方式、任务边界和协作模型：

- 多智能体：`AgentTool` / `SendMessageTool`
- 规划与流程：`EnterPlanMode` / `ExitPlanMode`
- 任务追踪：`TaskCreate` / `TaskGet` / `TaskUpdate` / `TaskList`
- 会话隔离：`EnterWorktree` / `ExitWorktree`
- 用户交互：`AskUserQuestion` / `CronCreate` / `CronDelete`

#### 3. 扩展桥接类工具

负责把 Claude Code 内部执行模型与外部能力连接起来：

- `MCPTool` / `ListMcpResourcesTool` / `ReadMcpResourceTool`
- `SkillTool`

这一层要点明一个核心判断：

> 协调类与桥接类工具的价值，不在于“替 AI 干活”，而在于“决定 AI 以什么方式干活”。

### 第三部分：权限检查链的完整流程

这是本次扩写的重点，建议用“链路分解 + 结论”写法。

建议链路如下：

1. LLM 输出 `tool_use`
2. 系统依据工具 schema 解析参数并归一化输入
3. 进入权限系统，根据工具名、参数内容、路径或命令文本匹配 allow / deny / ask 规则
4. Shell 类工具额外走危险命令静态分类器
5. 如果配置了 `permission_request` hook，则把工具请求转交给外部审批脚本
6. 得到 allow / reject / ask 的最终结果后才真正执行工具
7. 执行完成后，再根据 hook 配置触发 `post_tool_use` 等后置逻辑

这部分不需要完整搬运 `1.3.1` 的内容，但必须强调三点：

- 权限不是一个独立弹窗，而是嵌在工具执行链内部的必经步骤
- 权限规则与工具分类紧密绑定，`FileRead`、`Glob`、`Grep` 这类只读工具天然落在低风险区，而 `Bash`、`FileWrite`、`NotebookEdit` 这类高副作用工具会进入更严格的审批路径
- Shell 类工具因为风险外溢范围大，所以在规则匹配之外还要叠加静态危险性分析

建议在本段结束时给出一句分析性结论：

> 这说明 Claude Code 的“工具”从来不是普通函数调用，而是一个需要先通过策略判断才能被释放执行权的操作对象。

### 第四部分：大结果处理与临时文件机制

这一部分负责补足当前章节中完全缺失的“结果返回层”视角。

核心要讲清：Claude Code 不会把所有工具输出都原样塞回 `tool_result`，因为那样会立刻吞掉上下文预算。

建议写成三层处理模型：

1. **小结果**：直接内联返回，作为 `tool_result` 注入下一轮上下文
2. **中等结果**：截断后返回摘要，保留有限的可见窗口
3. **超大结果**：把完整输出写入临时文件或外部落点，只把摘要、路径、后续建议操作返回给模型

这里要解释两个“为什么”：

- **为什么需要临时文件**：因为完整输出仍可能对后续步骤有用，不能简单丢弃
- **为什么只回传摘要**：因为模型下一轮只需要知道“结果的大意”和“如何继续钻取”，并不需要一次吃下所有原始数据

建议用一句判断来提升抽象层级：

> 这意味着工具结果在 Claude Code 里不是“字符串返回值”，而是“受上下文预算约束的分层交付物”。

可补充的常见例子：

- 大型 `git diff`
- Shell 命令长输出
- 搜索结果命中过多
- Web 抓取内容过长

### 第五部分：工具并发安全控制策略

这一部分要和 `1.2.8` 做分工：本节讲“分类原则”，后者讲“执行引擎实现”。

建议组织方式：

1. Claude Code 不会对所有工具一视同仁地并发调度
2. 每个工具都有并发属性判断，核心是 `isConcurrencySafe`
3. 连续出现的并发安全工具可以合并成一个批次并行执行
4. 有副作用的工具必须拆成单独批次串行执行
5. 同批工具若出现失败，可通过中止机制取消兄弟任务，避免状态继续漂移

要明确点出只读与可写之间的差异：

- `GlobTool`、`GrepTool`、`FileRead` 这类读操作一般天然适合并发
- `FileEdit`、`FileWrite`、`BashTool` 因为会修改文件、进程或环境状态，必须被保守处理

这一节的关键结论建议写成：

> Claude Code 的并发不是“把多个工具一起跑”，而是把副作用隔离、状态一致性和响应延迟做权衡之后的受控并发。

### 收束段

章节结尾要把“分类、权限、结果裁剪、并发控制”四件事收束成一个统一认识。

推荐结论：

> 从架构上看，Claude Code 的工具体系真正特别的地方，不是工具数量，而是把能力暴露、权限控制、结果裁剪、并发调度、外部扩展统一收敛到了同一套工具抽象里。工具在这里不是 API 列表，而是 Agent Loop 可以安全编排的执行节点。

## 与其他章节的边界

为避免重复，扩写后的 `1.2.2` 应控制在“工具视角”的解释范围内：

- **与 `1.2.1 Agent Loop` 的边界**：这里只说明工具结果如何回流到下一轮，不展开 ReAct 主循环细节
- **与 `1.2.8 流式执行引擎` 的边界**：这里只说明为什么需要并发安全分类，不重复执行器状态机的完整实现
- **与 `1.3.1 权限系统` 的边界**：这里只讲工具调用时的权限链，不把权限模式、规则源叠加、bashClassifier 细节全部搬过来

这样可以让章节之间形成递进关系，而不是互相覆盖。

## 成稿标准

最终替换到 `1.2.2` 的正文应满足：

1. 开头有一个高质量总论，不再只是“工具分两类”
2. 至少完整覆盖用户点名的 4 个主题
3. 风格上能与 `1.2.1`、`1.2.5`、`1.3.1` 无缝衔接
4. 篇幅控制在中等扩写区间，不让本节喧宾夺主
5. 读者读完后能够形成一个清晰判断：Claude Code 的工具体系强，不是因为工具多，而是因为工具被放进了一条受控执行链里
